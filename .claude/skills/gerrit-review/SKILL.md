Fetch review feedback from a GerritHub change and present it clearly.

## Extracting the change number

The argument `$ARGUMENTS` is either:
- A bare change number (e.g. `1233340`)
- A GerritHub URL (e.g. `https://cue.gerrithub.io/c/cue-lang/cuelang.org/+/1233340`)

Extract the change number from whichever form is provided. For URLs the
change number is the final numeric path segment.

## Gerrit API base URL

The file `codereview.cfg` in the root of this repository on disk contains
a `gerrit:` line giving the project-scoped authenticated URL (e.g.
`https://cue.gerrithub.io/a/cue-lang/cuelang.org`). The REST API
endpoints use the **server root with `/a/` prefix** for authentication,
not the full project-scoped URL:

```
https://cue.gerrithub.io/a/changes/<NUMBER>/...
```

## Authentication

Always use `curl --netrc` for authentication. Credentials are stored in
`~/.netrc`.

## Gerrit JSON response format

All Gerrit REST API JSON responses are prefixed with `)]}'` on the first
line as an XSS protection. Always strip this before parsing. Use
`tail -c +5` to remove the 4-byte prefix:

```bash
curl --netrc -s '<URL>' | tail -c +5
```

## Fetching inline comments

```bash
curl --netrc -s 'https://cue.gerrithub.io/a/changes/<NUMBER>/comments' | tail -c +5
```

Returns a JSON object keyed by file path. Each comment object contains:
- `line` — the line number in the file
- `message` — the comment text
- `author.name` — who wrote it
- `unresolved` — whether it still needs attention
- `patch_set` — which patchset it was left on

## Fetching change details and top-level messages

```bash
curl --netrc -s 'https://cue.gerrithub.io/a/changes/<NUMBER>/detail' | tail -c +5
```

The `messages` array contains top-level change messages. Filter out
autogenerated messages (those with a `tag` field starting with
`autogenerated:`) to find human review comments.

## Presenting feedback

Display the comments grouped by file, showing:
1. The file path and line number
2. The comment text
3. Whether it is resolved or unresolved

Focus on **unresolved** comments as these need action.

When the file `/COMMIT_MSG` appears as a path, present those comments as
feedback on the **commit message** itself, not a source file. Line numbers
in `/COMMIT_MSG` correspond to the commit message text displayed in
Gerrit's diff view.

After listing all comments, summarise the overall feedback and suggest a
concrete plan for addressing each unresolved comment.
