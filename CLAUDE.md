# cuelang.org - Project Guide

This is the source repository for [cuelang.org](https://cuelang.org), the
home of the CUE language documentation and website.

## Mission

This repo serves two purposes:

1. **Content development** - authoring and maintaining documentation for
   cuelang.org
2. **Community support** - using the site's content as a knowledge base
   to answer questions from the CUE community on Slack, Discord, and
   GitHub Discussions

## Repository Structure

```
cuelang.org/
├── content/           # Source content (author-editable)
│   ├── docs/          # Documentation pages (Diataxis structure)
│   ├── examples/      # Standalone examples
│   └── site.cue       # Content-level CUE config
├── hugo/              # Hugo site (generated by preprocessor)
│   ├── content/en/    # Processed content (DO NOT edit directly)
│   ├── layouts/       # Hugo templates and shortcodes
│   ├── assets/        # SCSS, TypeScript, images
│   └── config/        # Hugo configuration
├── internal/          # Internal tooling
│   └── cmd/preprocessor/cmd/  # The preprocessor
├── _scripts/          # Build and development scripts
├── playground/        # CUE Playground app (/play)
├── site.cue           # Root site config (versions, defaults)
├── hugo.cue           # Hugo-specific config generation
└── site_tool.cue      # CUE tool for generating artifacts
```

## Documentation Structure (Diataxis)

The documentation at `content/docs/` follows the
[Diataxis](https://diataxis.fr/) framework, which organises content into
four categories based on user needs:

| Section | Diataxis Category | Purpose | Content Style |
|---------|------------------|---------|---------------|
| `introduction/` | — | Entry point for new users | Installation, "What is CUE?", getting started |
| `tour/` | — | Interactive language tour | Bite-sized pages showing CUE features |
| `tutorial/` | **Tutorials** (learning-oriented) | Practical lessons run on user's computer | Step-by-step, self-contained, teaches concepts through doing |
| `howto/` | **How-to Guides** (task-oriented) | Directions to achieve specific goals | Step-by-step instructions, assumes knowledge, goal-focused |
| `concept/` | **Concept Guides** (understanding-oriented) | Theory and foundations behind CUE | Explanatory, explores ideas and design decisions |
| `reference/` | **Reference** (information-oriented) | Language spec, CLI docs, glossary | Precise, complete, structured for lookup |
| `integration/` | — | CUE with other tools (Go, JSON, YAML, etc.) | Mixed tutorials/how-tos for specific ecosystems |
| `language-guide/` | — | Deep dive into language features | Detailed learning journey (in development) |
| `draft/` | — | Work-in-progress content | Not published to production |

### When deciding where new content belongs

- **"How do I...?"** → `howto/` (task-oriented, assumes context)
- **"Teach me about..."** → `tutorial/` (learning-oriented, builds from
  scratch)
- **"Why does CUE...?"** → `concept/` (understanding-oriented, explains
  design)
- **"What is the exact syntax for...?"** → `reference/` (lookup-oriented,
  precise)

## Content Authoring

### Preprocessor reference and examples

The processed output at `hugo/content/en/examples/cue/preprocessor/index.md`
is the authoritative reference for how preprocessor features work in
practice. Consult this file (and its source at
`content/examples/cue/preprocessor/en.md`) when writing content to see
working examples of `script`, `upload`, `code`, `#assert`, hidden blocks,
and other preprocessor features.

### Page structure

Each content page lives in its own directory under `content/docs/`:

```
content/docs/section/topic/
├── en.md           # Main content (markdown + preprocessor directives)
├── page.cue        # Page config (typically just a path reference)
└── gen_cache.cue   # Generated cache (DO NOT edit manually)
```

Branch pages (sections with children) use `_en.md` (underscore prefix).

### en.md format

```markdown
---
title: "Page Title"
weight: 10
tags:
  - cue-command
  - validation
---

Regular markdown content here.

{{{with code "en" "example-label"}}}
exec cue export input.cue
cmp stdout out
-- input.cue --
message: "hello"
-- out --
{
    "message": "hello"
}
{{{end}}}
```

### Preprocessor directives

Content uses `{{{` / `}}}` delimiters for preprocessor template syntax:

| Directive | Purpose |
|-----------|---------|
| `{{{with code "en" "label"}}}` | Code block - formats, validates, and renders CUE/other code |
| `{{{with script "en" "label"}}}` | Script block - runs commands in Docker, captures output |
| `{{{with _script_ "en" "label"}}}` | Hidden script - runs but doesn't render (setup commands) |
| `{{{with upload "en" "label"}}}` | Upload block - static file content in txtar format |
| `{{{with _upload_ "en" "label"}}}` | Hidden upload - file setup without rendering |
| `{{{uploaddir "path"}}}` | Upload a directory relative to page root |
| `{{{with step}}}` | Step container for numbered workflow steps |

Code and script blocks use
[testscript](https://pkg.go.dev/github.com/rogpeppe/go-internal/testscript)
format internally. Common testscript commands: `exec`, `cmp`, `cp`,
`mkdir`, `env`, `! exec` (expect failure), `stderr`/`stdout` (match
output).

Special tags within blocks:
- `#norun` - don't execute (display only)
- `#nofmt` - don't format
- `#ellipsis` - allow `...` in output matching
- `#location left right` - side-by-side display
- `#force` - overwrite existing files
- `#assert` - assert file contents match

### page.cue files

Minimal CUE files that wire the page into the site config hierarchy:

```cue
package site

content: docs: section: topic: page: _
```

The `_` means "accept all defaults from parent config". Page-specific
overrides can set `leftDelim`, `rightDelim`, `vars`, `comparators`, etc.

### gen_cache.cue files

Auto-generated by the preprocessor. Store hashes of code/script outputs
to avoid re-running expensive operations. **Never edit these manually.**

### Hugo shortcodes

Common shortcodes used in content (these use Hugo `{{<` / `>}}`
delimiters, not preprocessor delimiters):

- `{{< relref "docs/path/to/page" >}}` - internal cross-references
- `{{< cards >}}` / `{{< card >}}` - card grid layout
- `{{< info >}}` / `{{< warning >}}` / `{{< caution >}}` - admonition
  boxes
- `{{< code-tabs >}}` - tabbed code display (generated by preprocessor)
- `{{< step >}}` - numbered step (generated by preprocessor)
- `{{< tag "name" >}}` - content tag
- `{{< search >}}` - search interface

## The Preprocessor

The preprocessor (`internal/cmd/preprocessor/cmd/`) transforms content
from `content/` into Hugo-ready pages in `hugo/content/en/`.

### What it does

1. **Parses** markdown files, extracting YAML frontmatter and template
   directives
2. **Validates** node syntax, label uniqueness, and configuration
3. **Executes** code blocks (formatting, CUE evaluation) and scripts
   (in Docker containers)
4. **Caches** results in `gen_cache.cue` files
5. **Transforms** preprocessor directives into Hugo shortcodes
6. **Writes** processed content to `hugo/content/en/`

### Running the preprocessor

```bash
# Full build (preprocessor + Hugo)
./_scripts/build.bash

# Local development with live-reload
./_scripts/serve.bash

# Run preprocessor only
./_scripts/runPreprocessor.bash execute

# With flags
./_scripts/serve.bash --update  # Write changes back to source files
```

### Key preprocessor flags

- `--debug=<category>` - Verbose logging (general, cache, script, code,
  formatting, fsnotify, hugo)
- `--update` - Write changes back into source files
- `--skipcache` - Force re-run all code/scripts (ignore cache)
- `--readonlycache` - Fail on cache miss (used in CI)
- `--serve` - Watch for changes and re-execute
- `--mkdocs-output=<dir>` - Output in mkdocs format instead of Hugo

### Cache management workflow

When making preprocessor changes that affect cache generation, use a
two-commit pattern:

1. **First commit** - the actual changes, with
   `Preprocessor-No-Write-Cache: true` trailer:
   ```
   preprocessor: your change description

   Signed-off-by: Your Name <your@email.com>
   Preprocessor-No-Write-Cache: true
   ```

2. **Second commit** - regenerate cache files:
   ```bash
   ./_scripts/runPreprocessor.bash execute
   git add content/docs/*/gen_cache.cue
   git commit -m "content: regenerate cache"
   ```

## Testing the Preprocessor

Tests live in `internal/cmd/preprocessor/cmd/testdata/` using testscript
format (`.txtar` files).

### Naming convention

- `execute_<scenario>.txtar` - successful execution tests
- `execute_err_<scenario>.txtar` - tests expecting errors

### Running tests

```bash
# All preprocessor tests
go test ./internal/cmd/preprocessor/cmd/

# Specific test
go test -run TestScript/<test_name> ./internal/cmd/preprocessor/cmd/

# Verbose output
go test -run TestScript/<test_name> -v ./internal/cmd/preprocessor/cmd/

# Update golden files
CUE_UPDATE=1 go test ./internal/cmd/preprocessor/cmd/
```

### Test file structure

```
# Comment describing the test

unquote content/example/en.md
mkdir output

! exec preprocessor execute --mkdocs-output=output
stderr 'expected error pattern'

-- content/example/site.cue --
package site
...

-- content/example/en.md --
>---
>title: Test
>---
>
>{{{with code "en" "test"}}}
>{{{end}}}
```

Use `>` prefix + `unquote` for files containing frontmatter or template
syntax. Use `!` prefix for commands expected to fail. Use `stderr`/`stdout`
for output pattern matching.

## Community Support Workflow

### Knowledge base

The site content at `content/docs/` is the primary knowledge base for
answering community questions. When responding to questions:

1. **Search existing content first** - check if the answer is already
   documented on cuelang.org. Grep across `content/docs/` for relevant
   keywords.
2. **Reference specific pages** - link to the published page URL at
   `https://cuelang.org/docs/...` when the answer exists.
3. **Identify content gaps** - if no existing content covers the
   question, note this as a potential new page or improvement.

### Channels

- **Slack** - CUE community Slack workspace
- **Discord** - CUE Discord server
- **GitHub Discussions** -
  https://github.com/cue-lang/cue/discussions

### Drafting responses

When drafting responses to community questions:

- Output responses as **raw markdown** suitable for copy-paste into the
  target platform
- Reference existing cuelang.org pages with full URLs where relevant
- Include working CUE examples where helpful
- Keep answers concise but complete
- If the question reveals a documentation gap, mention this separately

### Content gap tracking

Content that should exist on cuelang.org but doesn't yet is tracked at:
https://github.com/cue-lang/docs-and-content/issues

Issues in that repo use labels to indicate the Diataxis category:
`howto`, `tutorial`, `concept`, `reference`, etc. When working on an
issue:

- **If a label exists** - assess whether it's the right Diataxis
  category for the content. A task-oriented "How do I X?" piece belongs
  in `howto/`, not `tutorial/`, even if labelled otherwise. Flag any
  mismatch.
- **If no label exists** - suggest the most appropriate Diataxis
  category based on the content described in the issue, using the
  guidelines in the "Documentation Structure" section above.

When a community question reveals a gap:
1. Check if an issue already exists in docs-and-content
2. If not, note the gap for potential issue creation, including the
   suggested Diataxis category
3. The eventual fix is to create the content on cuelang.org so future
   users can self-serve

### Using GitHub Discussions

Use `gh` CLI to interact with GitHub Discussions:

```bash
# List recent discussions
gh api repos/cue-lang/cue/discussions --jq '.[] | {number, title}'

# View a specific discussion
gh api repos/cue-lang/cue/discussions/<number>
```

## Site Configuration

### site.cue (root)

Defines versions, environment defaults, and content configuration:

- `versions.cue.latest.v` - current CUE version documented by the site
- `versions.go` - Go version used in examples
- `baseEnv` - environment variables for code/script execution
- `_contentDefaults` - default page settings (delimiters, sanitisers,
  comparators)

### Version updates

To update the documented CUE version, modify `versions.cue.latest.v` in
`site.cue` and follow the cache regeneration workflow.

## Commit Message Guidelines

- Use prefix matching the area changed: `preprocessor:`, `content:`,
  `site:`, etc.
- Follow the CUE project convention: first line completes "This change
  modifies CUE to ___"
- Include `Signed-off-by` line (`git commit -s`)
- Always use `--no-gpg-sign` (or `-c commit.gpgsign=false`) to skip
  GPG signing, which requires interactive input
- No AI authorship attribution in commit messages
- All commits must include a `Change-Id` line (added automatically by
  `git codereview change`)

### Editing commits in a stack

```bash
# Rebase from branchpoint
GIT_SEQUENCE_EDITOR="sed -i 'Ns/^pick/edit/'" git rebase -i $(git codereview branchpoint)
```

Where `N` is the line number of the commit to edit.
